## Project Requirements Document

Version: 1.0

# 1. Project Overview

## 1.1 Objective

Develop a dynamic proxy pool management system based on:

- Oracle VPS
- frps
- GitHub Actions Runner
- frpc SOCKS5 plugin

The system uses GitHub Actions Hosted Runner as temporary proxy exit nodes.

Each Runner establishes a reverse connection to Oracle VPS through frpc.

Oracle VPS provides centralized proxy management:

- Allocate proxy slots
- Manage remote ports
- Create and trigger GitHub workflows
- Monitor workflow lifecycle
- Monitor proxy availability
- Recycle proxy resources

# 3. Core Design Principles


## 3.1 Fixed Proxy Slot Model

The system uses fixed proxy slots.

A proxy slot represents:

One workflow file
=
One fixed remotePort
=
One proxy resource
Example:
proxy-slot-10801.yml

binds to:

remotePort = 10801


The workflow file is created once and reused.

The system does NOT create a new workflow file every time a proxy is requested.


---

## 3.2 Workflow and Port Binding


Each workflow has a permanent port mapping.


Example:
proxy-slot-10801.yml
|
|
+---- remotePort=10801

proxy-slot-10802.yml
|
|
+---- remotePort=10802



This mapping never changes.

# 8. State Definitions


## free


Meaning:

- Slot available
- No active workflow
- Port can be reused


---

## pending


Meaning:

Workflow has been triggered.

GitHub status:


queued



The Runner has not connected yet.


---

## running


Conditions:

ALL must be true:


1. GitHub workflow:


in_progress



2. frps connection exists


3. SOCKS5 health check succeeds


---

## failed


Possible reasons:


- Workflow failed
- Runner startup failed
- frpc failed
- Timeout
- GitHub unavailable


---

## stopping


Proxy is being recycled.


Process:



Cancel workflow

    |

Wait Runner exit

    |

Check frp disconnect

    |

Release slot

18. Development Restrictions

The implementation MUST NOT:

Create unlimited workflow files dynamically
Allow multiple active runs using the same port
Allocate ports without locking
Depend on manual port management
Delete workflow files during normal recycling
19. Development Phases

Phase 3

Implement:

Automatic recovery
Proxy lifecycle management
20. Acceptance Criteria

The system must:

Support multiple simultaneous proxy nodes
Guarantee no remotePort conflicts
Reuse proxy slots
Monitor GitHub workflow status
Detect pending/running/failed states
Automatically recover failed nodes
Allow proxy deletion
Allow port reuse after cleanup
Maintain persistent proxy state after restart
