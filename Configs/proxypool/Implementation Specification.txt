GitHub Actions Ephemeral Proxy Pool Controller
Project Requirements and Implementation Specification

Version: 5.0

Target Implementation: Cline

Architecture: Oracle VPS + GitHub Actions Hosted Runner + frps + frpc + Dynamic Proxy Session

1. Project Overview

本项目用于构建一个基于 GitHub Actions Hosted Runner 的动态临时代理池。

系统利用 GitHub Actions Hosted Runner 作为临时代理出口节点。

Oracle VPS 作为中央控制面和代理池管理中心。

系统主要组件：

Oracle VPS
├── Proxy Controller
├── SQLite Database
├── frps
└── Proxy Health Checker

GitHub
└── GitHub Actions

GitHub Actions Hosted Runner
├── Runner Bootstrap Agent
├── frpc
└── SOCKS5 Proxy

系统的核心目标：

Oracle 动态创建 GitHub Workflow。
每个 Workflow Run 只包含一个 Job。
每个 Job 使用一个 GitHub Hosted Runner。
Runner 启动后主动向 Oracle 注册。
Runner 自己获取公网出口 IP。
Oracle 动态分配 Remote Port。
Runner 动态生成 frpc.toml。
Runner 通过 FRP 反向连接 Oracle。
Oracle 通过 Remote Port 提供代理访问入口。
Oracle 对 Proxy 执行端到端健康检查。
Proxy 异常时自动终止对应 Workflow。
Oracle 不直接控制 Runner 停止 frpc。
Proxy 停止通过 GitHub API Cancel Workflow Run 实现。
Workflow 结束后 Runner 和 frpc 自动退出。
Oracle 确认 FRP Tunnel 断开后回收端口。
需要新 IP 时创建新的 Workflow，而不是 Rotate 原有 Runner。
2. Fundamental Architecture

系统采用严格的一对一映射：

1 Workflow Run
       │
       │ 1:1
       ▼
1 Job
       │
       │ 1:1
       ▼
1 Runner
       │
       │ 1:1
       ▼
1 Proxy Session
       │
       │ 1:1
       ▼
1 Remote Port
       │
       │
       ▼
1 Proxy

因此：

1 Workflow Run = 1 Proxy

这是本系统最重要的架构原则。

3. Single-Job Workflow Requirement

所有 Proxy Workflow 必须是单 Job Workflow。

正确：

jobs:
  proxy:
    runs-on: ubuntu-latest

不允许：

jobs:
  proxy1:
    runs-on: ubuntu-latest

  proxy2:
    runs-on: ubuntu-latest

原因：

多 Job Workflow：

Workflow Run
├── Job A → Runner A → Proxy A
├── Job B → Runner B → Proxy B
└── Job C → Runner C → Proxy C

取消 Workflow Run 会导致多个 Runner 同时停止。

本项目必须保持：

Workflow Run
└── Job
    └── Runner
        └── Proxy

这样：

1 Run
=
1 Job
=
1 Runner
=
1 Proxy
4. Proxy Lifecycle Equals Workflow Lifecycle

Proxy 不拥有独立于 Workflow 的生命周期。

生命周期关系：

Workflow Created
    =
Proxy Created

Workflow Running
    =
Proxy Starting / Running

Workflow Cancelled
    =
Proxy Stopping

Workflow Completed
    =
Proxy Stopped

因此 Oracle 不需要自己维护一个独立的 Runner 进程控制机制。

10. Runner Must Report Job ID

Oracle 不应该根据 GitHub API 返回的 Job 列表猜测 Runner 对应关系。

Runner 启动后必须主动注册：

Runner
  │
  ├── run_id
  ├── job_id
  ├── runner_id
  └── external_ip
        │
        ▼
Oracle

这样可以避免：

多个 Workflow 同时运行
GitHub API 查询延迟
Job 创建时间相近
Runner 启动顺序不确定

导致错误绑定。

11. Runner External IP

Runner 启动后必须自行获取公网出口 IP：

curl -4 -fsSL https://ipconfig.io

例如：

1.2.3.4

Runner 将其作为：

external_ip

注册给 Oracle。

12. External IP Must Bypass Proxy

获取 external_ip 时，Runner 必须直接访问：

https://ipconfig.io

不得经过：

SOCKS5
frpc
代理环境变量

必须确保：

Runner
  │
  │ Direct HTTPS
  ▼
ipconfig.io

而不是：

Runner
  │
  ▼
SOCKS5
  │
  ▼
ipconfig.io

这样 Oracle 得到的 external_ip 才是 Runner 原始出口 IP。

Runner → Oracle Control Plane 不需要认证。

以下接口不要求：

Bearer Token
GitHub PAT
Runner Token

例如：

POST /api/v1/sessions/register

POST /api/v1/sessions/{session_id}/ready

但是 Oracle 必须验证：

14. Runner Registration

接口：

POST /api/v1/sessions/register

Request：

{
  "run_id": "123456789",
  "job_id": "987654321",
  "runner_id": "runner-abc123",
  "hostname": "runner-host",
  "external_ip": "1.2.3.4"
}

Oracle 执行：

1. 验证 run_id
2. 验证 job_id
3. 验证 run / job 状态
4. 防止重复注册
5. 创建 Proxy Session
6. 分配 Remote Port
7. 保存 external_ip
8. 返回 FRP 配置

Response：

{
  "session_id": "sess-a8f32d",
  "remote_port": 10003,
  "frps_server": "oracle.example.com",
  "frps_port": 7000
}
15. Duplicate Registration Protection

必须保证：

一个 run_id
    最多一个 Active Session

一个 job_id
    最多一个 Active Session

一个 runner_id
    最多一个 Active Session

如果重复注册：

run_id = 12345

已经存在：

则不得创建：

22. WAITING_FOR_RUNNER

GitHub Workflow 已创建，但 Runner 尚未开始执行。

Oracle 定期查询 GitHub API。

建议：

Polling Interval = 5 秒

检查：

Workflow Run Status
Job Status

如果超过：

RUNNER_START_TIMEOUT = 120 秒

则：

WAITING_FOR_RUNNER
    ↓
STARTUP_TIMEOUT
    ↓
Cancel Workflow
    ↓
FAILED

典型场景：

    ↓

    ↓
GitHub Run Created
    ↓

    ↓

    ↓

29. End-to-End Proxy Health Check

Oracle 必须对 Proxy 执行真正的端到端检查。

检查路径：

Oracle
   │
   │ SOCKS5
   ▼
Remote Port
   │
   ▼
FRP Tunnel
   │
   ▼
Runner SOCKS5
   │
   ▼
Internet

例如：

curl \
  --socks5-hostname 127.0.0.1:10001 \
  https://ipconfig.io

或者：

curl \
  --socks5-hostname 127.0.0.1:10001 \
  https://api.ipify.org

Oracle 获取：

Observed IP

然后比较：

Observed IP
    ==
external_ip

如果一致：

Proxy HEALTHY
30. Why End-to-End Health Check Is Required

以下状态不能直接认为 Proxy 正常：

Runner Registered

因为这只能证明：

Runner → Oracle

正常。

以下状态也不能直接认为 Proxy 正常：

FRP Tunnel Connected

因为这只能证明：

frpc → frps

正常。

真正可用必须验证：

Oracle Remote Port
    ↓
FRP
    ↓
Runner SOCKS5
    ↓
Internet

因此：

Workflow OK
+
Runner OK
+
Control Plane OK
+
FRP OK
+
SOCKS5 OK
+
Internet OK
+
Observed IP == external_ip
=
HEALTHY
31. HEALTHY

Proxy 满足：

Workflow 正常
Job 正常
Runner 已注册
Heartbeat 正常
FRP Tunnel 正常
SOCKS5 正常
Internet 正常
Observed IP == external_ip

状态：

HEALTHY

此时 Oracle 才允许 Proxy 进入正式 Proxy Pool。

37. FRP Tunnel Disconnect

当 Proxy 被取消：

Cancel Workflow
    ↓
Runner Exit
    ↓
frpc Exit
    ↓
FRP Disconnect

Oracle 必须确认 Tunnel 实际断开。

不能：

Cancel Workflow
    ↓
立即 Port → FREE

因为 GitHub Cancel 不一定立即导致 Runner 进程终止。

38. Port Release

Remote Port 只有满足以下条件才能释放：

1. Workflow 已结束
2. Runner Heartbeat 已停止
3. FRP Tunnel 已断开
4. frps 不再持有该 Remote Port

然后：

Session → RELEASED
Port → FREE
39. FRP Port Release Delay

Workflow Cancel 后，FRP 端口可能不会立即释放。

允许：

最大等待 30 秒

检查：

每 1 秒

流程：

Cancel Workflow
    ↓
Wait
    ↓
Check Tunnel
    ↓
Disconnected?
    ├── Yes → Release Port
    └── No  → Continue Waiting

30 秒后仍未释放：

CLEANUP_REQUIRED

进入异常清理流程。

41. New IP / Rotation Model

本系统不实现传统意义上的：

Rotate Existing Runner

因为：

1 Workflow Run = 1 Proxy

需要新 IP 时直接创建新 Workflow。

例如：

Proxy A
external_ip = 1.1.1.1

需要新 IP：

Create Workflow B
    ↓
Runner B
    ↓
external_ip = 2.2.2.2
    ↓
Proxy B HEALTHY

然后：

Switch Traffic
    ↓
Cancel Workflow A

因此：

新建 Workflow
=
新 Runner
=
新公网 IP
=
新 Proxy
42. Zero-Downtime Replacement

旧 Proxy：

Proxy A
run_id = 10001
port = 10001

HEALTHY

创建新 Proxy：

run_id = 10002

Traffic
   ↓

可以实现零停机换 IP。

55. Final System Rules

Cline 实现时必须遵循以下规则：

1. Workflow 必须是单 Job。

2. 1 Run = 1 Job。

3. 1 Job = 1 Runner。

4. 1 Runner = 1 Proxy Session。

5. 1 Session = 1 Remote Port。

6. Workflow 不绑定固定 Oracle 端口。

7. Runner 启动后主动注册 Oracle。

8. Runner 自己获取 external_ip。

9. external_ip 必须直接 curl ipconfig.io 获取。

10. 获取 external_ip 不得经过 SOCKS5。

11. Runner → Oracle 不需要认证。

12. Oracle 根据 run_id / job_id 建立映射。

13. Oracle 动态分配 Remote Port。

14. Runner 动态生成 frpc.toml。

15. Oracle 必须执行 FRP Tunnel 检查。

16. Oracle 必须执行端到端 SOCKS5 健康检查。

17. Proxy 健康检查必须验证实际出口 IP。

18. Observed IP 必须与 Runner external_ip 一致。

19. Workflow 启动超时必须自动取消。

20. Runner 注册超时必须自动取消。

21. FRP Tunnel 连接超时必须自动取消。

22. Proxy 连续健康检查失败必须自动取消。

23. Heartbeat 超时必须自动取消或进入清理。

24. 停止 Proxy 时必须通过 GitHub API Cancel Workflow Run。

25. Oracle 不通知 Runner 删除 frpc 配置。

26. Oracle 不实现 Runner Release API。

27. Runner 随 Workflow / Job 终止。

28. frpc 随 Runner 终止。

29. Remote Port 必须等待 FRP Tunnel 实际断开后释放。

30. 新 IP 通过创建新的 Workflow Run 获取。

31. 不实现传统 Rotate Runner。

32. 可以通过新旧 Proxy 并行实现零停机换 IP。

33. Proxy 对外管理标识使用 run_id。

34. Oracle 内部 Session 标识使用 session_id。

35. 所有状态变化必须写入 SQLite。

36. 所有超时必须有明确状态和清理流程。

37. 所有异常最终必须进入 FAILED / CLEANUP / RELEASED 闭环。
